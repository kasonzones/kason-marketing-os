# /流失预防 (Churn Prevention)

> 🛡️ 设计取消挽留流程、优惠留存方案和催款策略，把流失用户变成终身客户

## 触发词

`/流失预防` `/churn-prevent`

## 功能概述

双轨策略覆盖两类流失场景：主动流失（用户主动取消）和被动流失（支付失败导致的非自愿流失）。融合支付平台 API，构建全链路的防流失防线。

## 主动流失：取消挽留流程 (Cancel Flows)

### 挽留阶梯（从轻到重）

| 层级 | 策略 | 触发条件 | 预期留存率 |
|------|------|---------|-----------|
| L1 🟢 | 暂停账号 30-90 天 | 用户点击取消 | 15-25% |
| L2 🟡 | 降级到免费版 | 暂停被拒绝 | 10-20% |
| L3 🟠 | 月度 50% 折扣 × 3 个月 | 降级被拒绝 | 20-35% |
| L4 🔴 | 年度 30% 折扣 | 月度折扣被拒 | 15-25% |
| L5 💔 | 最终挽留页 + 反馈收集 | 所有优惠被拒 | — |

### 取消页面最佳实践

```yaml
最佳实践清单:
  ✅ 让取消流程不超过 3 步
  ✅ 每一步展示用户将失去的价值
  ✅ 使用具体数据（"你将失去 12 个活跃项目"）
  ✅ 提供"暂停"替代"取消"
  ✅ 收集取消原因（至少 5 个预设选项 + 自由输入）
  ✅ A/B 测试挽留方案排列顺序
  ❌ 不要隐藏取消按钮
  ❌ 不要强制用户联系客服才能取消
  ❌ 不要在取消流程中加太多挽留步骤（超过 3 步 = 体验灾难）
```

## 被动流失：支付失败恢复 (Dunning Management)

### 智能催款策略

| 事件 | 时间线 | 动作 | 渠道 |
|------|--------|------|------|
| 💳 首次支付失败 | T+0 | 自动重试（智能路由） | Stripe/Chargebee |
| 📧 首次提醒 | T+1 | 支付方式更新提醒 | 邮件 |
| 📱 二次提醒 | T+3 | 应用内通知 + 邮件 | App + Email |
| 📞 人工介入 | T+7 | 客服联系（高价值客户） | 邮件 + 电话 |
| ⚠️ 最后通知 | T+14 | 服务即将暂停警告 | 邮件 + 短信 |
| 🛑 暂停服务 | T+21 | 暂停但保留数据 | 全渠道通知 |
| 🗑️ 数据删除 | T+90 | 最终清理 | 邮件确认 |

### 支付平台集成

#### Stripe 集成

```javascript
// Stripe 智能重试配置
{
  "automatic_payment_methods": { "enabled": true },
  "payment_method_options": {
    "card": {
      "network_preference": { "type": "automatic" }
    }
  }
}

// 关键事件 Webhook
events: [
  "invoice.payment_failed",        // → 触发催款序列
  "invoice.payment_succeeded",     // → 停止催款序列
  "customer.subscription.updated", // → 记录套餐变更
  "payment_method.detached"        // → 提醒用户更新卡
]
```

#### Chargebee 集成

```yaml
Chargebee 催款配置:
  dunning_attempts: 5
  retry_schedule: [3, 7, 14, 21, 28]  # 天
  smart_retry: true                     # 基于银行响应码智能重试
  card_update_reminder: true            # 自动发送卡更新链接
  grace_period_days: 10                 # 宽限期
```

#### Paddle 集成

```yaml
Paddle 全球支付恢复:
  localized_dunning: true    # 根据地区自动调整催款策略
  tax_compliance: auto       # 自动处理全球税务
  retry_logic: intelligent   # Paddle 内置智能重试
  currency_optimization: true # 多币种优化
```

## AutoWash Pro X1 示例

### 示例 1：SaaS 主动取消防御

```yaml
场景: AutoWash Pro X1 云管理平台 SaaS 订阅取消
触发: 用户在后台点击"取消订阅"

取消流程设计:
  步骤1 - 价值提醒页:
    展示内容:
      - 📊 "本月你通过 AutoWash Pro X1 完成了 847 次洗车服务"
      - 📈 "相比上月增长了 12.3%"
      - ⏱️ "智能调度为你节省了 32 小时人工"
    操作选项:
      - [继续取消]
      - [我再想想] ← 主 CTA（大按钮、品牌色）

  步骤2 - 暂停替代:
    标题: "先休息一下？"
    选项:
      - 🟢 暂停 30 天（免费保留所有数据）
      - 🟢 暂停 90 天（免费保留所有数据）
      - 🔴 仍然取消

  步骤3 - 优惠留存:
    标题: "我们不想失去你"
    方案A: 下 3 个月享受 40% 折扣
    方案B: 切换到年度计划省 25%
    方案C: 切换到轻量版（保留核心功能，$9/月）

  步骤4 - 反馈收集:
    取消原因:
      □ 价格太贵
      □ 功能不满足需求
      □ 使用频率低
      □ 竞品更好
      □ 不再需要
      □ 其他: _____

预期效果:
  - 挽留率目标: 35-45%
  - 收集取消原因覆盖: 90%+
  - NPS 在取消流程后 30 天回访
```

### 示例 2：支付失败恢复流程

```yaml
场景: AutoWash Pro X1 年度客户信用卡过期
触发: Stripe invoice.payment_failed webhook

Dunning 序列:
  Day 0 (失败即时):
    ├── Stripe 自动重试 3 次（间隔 4 小时）
    └── 系统记录: "payment_failed_attempt_1"

  Day 1 (首次提醒):
    📧 邮件主题: "AutoWash Pro X1 — 支付方式需要更新"
    内容:
      - 🫂 友好的语气，不是威胁
      - 🔗 一键更新支付方式链接（Stripe Customer Portal）
      - 📞 客服联系方式
      - ⏰ "你有 14 天时间更新，期间服务不受影响"

  Day 3 (二次提醒):
    📱 应用内通知: "你的支付方式需要更新才能继续享受服务"
    📧 邮件: 重申 + 添加"常见支付问题"FAQ 链接

  Day 7 (人工介入 - 仅年付 $1000+ 客户):
    📞 客户成功经理致电
    话术: "只是想确认你收到了支付更新的提醒，有什么我可以帮你的？"

  Day 14 (最后通知):
    📧 邮件主题: "最后提醒 — AutoWash Pro X1 账户即将暂停"
    内容:
      - ⚠️ 明确后果："3 天后服务将暂停"
      - 🔗 更新链接
      - 💬 在线客服直达

  Day 21 (服务暂停):
    暂停服务，但保留全部数据
    登录后显示: "更新支付方式即可恢复访问"

挽回率目标:
  - Day 1-3: 60-70% 恢复率
  - Day 4-14: 额外 15-20%
  - Day 14-21: 额外 5-10%
  - 总目标恢复率: 85%+
```

```yaml
预警信号:
  - 🚨 支持工单中"取消""退款""太贵"关键词突增
  - 🚨 登录频率连续下降 2 周
  - 🚨 核心功能使用率持续走低
  - 🚨 NPS 评分连续下滑
```

---

> 💡 **核心原则**: 挽留不是强留——挽留的核心是帮助用户重新发现价值。如果用户确实不适合，体面的离开比痛苦的挽留更能为未来赢回机会。
