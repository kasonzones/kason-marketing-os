# /AB测试 (A/B Test)

> 🧪 规划、设计和实施 A/B 实验，用数据驱动产品决策

## 触发词

`/AB测试` `/ab-test`

## 功能概述

从实验假设到统计分析的全流程支持。包含 ICE 优先级框架、假设模板、样本量计算器、实验设计模板和结果分析指南，让每一次产品变更都有数据支撑。

## ICE 优先级框架

用于在多个实验想法中选择最值得投入的：

```
ICE Score = Impact × Confidence × Ease
              │         │          │
              ▼         ▼          ▼
          影响力    信心度    实施难度
          (1-10)   (1-10)    (1-10)
```

| 维度 | 说明 | 1 分 | 10 分 |
|------|------|------|-------|
| 🎯 Impact (影响力) | 实验成功对核心指标的影响 | 几乎无影响 | 改变游戏规则 |
| 🔬 Confidence (信心度) | 有多确信实验结果会好 | 纯猜测 | 有强数据/案例支撑 |
| ⚡ Ease (实施难度) | 实施成本（时间/资源/技术） | 需要 1 个月 | 10 分钟可上线 |

```yaml
ICE 使用示例:
  实验 A: 修改注册按钮颜色
    I=3, C=5, E=9 → ICE = 135

  实验 B: 新用户引导流程重构
    I=9, C=7, E=3 → ICE = 189

  实验 C: 增加社交登录选项
    I=7, C=8, E=6 → ICE = 336 ← 最高优先级！

  结论: 优先执行实验 C
```

## 假设框架

### 标准假设模板

```
如果 [我们做出这个改变]
那么 [我们预测会产生这个结果]
因为 [基于这个理由/数据]
用 [这个指标] 来衡量
```

### 假设示例

```yaml
好假设 ✅:
  如果我们在注册页面减少到 3 个必填字段
  那么注册完成率会提升
  因为减少了用户填写负担，降低流失
  用注册完成率 (signup_complete / signup_start) 来衡量

差假设 ❌:
  如果我们改进注册页面
  那么用户会更多
  因为体验更好
  （没有具体指标、没有量化预期、没有数据依据）

优秀假设 🌟:
  如果我们在注册页首屏增加 3 个社交证明元素
  那么注册完成率会从当前 12% 提升到 16% 以上
  因为 Heatmap 显示 68% 用户在第一屏就离开了；
  社交证明可以减少对未知产品的犹豫
  用注册完成率来衡量，且已预先计算样本量需求为 2000 用户/变体
```

## 样本量计算

### 快速参考表

| 基线转化率 | 期望提升 | 最小样本量/变体 | 推荐实验时长 |
|-----------|---------|---------------|-------------|
| 5% | +20% (5%→6%) | 8,500 | 14-21 天 |
| 5% | +50% (5%→7.5%) | 1,500 | 7-14 天 |
| 10% | +10% (10%→11%) | 14,000 | 21-28 天 |
| 10% | +20% (10%→12%) | 3,800 | 14-21 天 |
| 10% | +50% (10%→15%) | 700 | 7-10 天 |
| 20% | +10% (20%→22%) | 11,000 | 21-28 天 |
| 20% | +20% (20%→24%) | 2,900 | 14-21 天 |
| 50% | +10% (50%→55%) | 4,100 | 14-21 天 |

*假设: α=0.05, β=0.20 (80% 统计功效), 双侧检验

### 样本量影响因素

```
所需样本量 ∝
    1
  ──────────
  (效果量)²
  
即: 期望效果越小 → 需要样本量越大（平方关系）

影响样本量的 4 个参数:
  1. 基线转化率 (Baseline Rate) — 越低，样本量越大
  2. 最小可检测效果 (MDE) — 越小，样本量越大
  3. 显著性水平 (α) — 通常 0.05，越小样本量越大
  4. 统计功效 (1-β) — 通常 0.80，越高样本量越大
```

### 常见实验类型

| 类型 | 适用场景 | 注意事项 |
|------|---------|---------|
| 🅰️/🅱️ A/B Test | 单变量对比 | 确保变体间隔离 |
| 🅼🆅 Multivariate | 多变量组合优化 | 需要更大样本量 |
| 🔀 Split URL | 不同 URL/页面 | 注意 SEO 影响 |
| 🔄 Multi-Armed Bandit | 自动调节流量分配 | 需要算法支持 |
| ⏰ Before/After | 大改动（无法 A/B） | 注意外部因素干扰 |
| 🌐 Geo Test | 按地区分流量 | 确认地区间可比性 |

## 结果分析

### 统计显著性判断

```yaml
决策矩阵:
  p < 0.05 + 效果为正 + 护栏正常 → ✅ 发布
  p < 0.05 + 效果为正 + 护栏恶化 → ⚠️ 需权衡
  p < 0.05 + 效果为负 → ❌ 放弃
  p ≥ 0.05 + 样本量足够 → ❌ 无显著差异（放弃）
  p ≥ 0.05 + 样本量不足 → 🔄 延长实验或增加流量

常见陷阱:
  ❌ 窥视偏差 (Peeking): 每天看结果，看到显著就停 → 假阳性
  ❌ 多重比较: 同时看 10 个指标，找到 1 个显著就宣布胜利
  ❌ 新奇效应: 实验结果好只是因为"新东西"→ 观察至少 2 周
  ❌ 选择偏差: 参与实验的用户不等于全部用户
  ❌ 忽略分段: 总体没效果，但某个用户群效果显著
```

## AutoWash Pro X1 示例

### 示例 1：注册页优化实验

```yaml
实验名称: 注册流程从 5 步改为 3 步
实验 ID: EXP-20260801-001
实验类型: A/B Test
负责人: 增长团队

问题定义:
  当前注册流程有 5 步（邮箱→密码→姓名→公司→电话），
  数据漏斗显示从 Step 2→3 流失 25%，Step 3→4 流失 18%。
  总注册完成率仅 8.3%，行业基准为 15%。
  Heatmap 显示第 3 步（姓名）和第 5 步（电话）的退出率最高。

假设:
  如果我们将注册流程从 5 步缩减到 3 步（邮箱→密码→公司名）
  那么注册完成率会从 8.3% 提升到 12% 以上
  因为减少了非必填字段的摩擦，用户无需在初始阶段提供电话和姓名
  用 signup_complete / signup_start 来衡量

实验设计:
  控制组 A (50%): 现有 5 步注册流程
  变体组 B (50%): 新 3 步注册流程
    Step 1: 邮箱 + 密码（合并）
    Step 2: 公司名称（必填）
    Step 3: 欢迎页 → 引导至连接设备
    姓名 + 电话 → 移至设备连接后可选填写

目标指标:
  主要指标: 注册完成率 (signup_complete / signup_start)
  次要指标:
    - 注册完成时间
    - Day 7 设备连接率
    - Day 30 留存率
  护栏指标:
    - 垃圾注册率（不能上升超过 20%）
    - 客服工单量（不能上升超过 10%）
    - 用户信息完整度（后续补填率）

样本量与时长:
  基线转化率: 8.3%
  期望最小提升: +45% (8.3% → 12%)
  所需样本量: ~1,400 / 变体
  日访问量: ~400
  预计时长: 10-14 天

成功标准:
  ✅ 注册完成率提升 ≥ 30% (8.3% → 10.8%+) 且 p < 0.05
  ✅ 护栏指标无显著恶化

实施:
  技术方案: Feature Flag (LaunchDarkly)
  Flag Key: "signup-3step-v1"
  埋点: 现有事件不变，新增 signup_step_abandoned

预期结果 (假设):
  控制组 A: 8.4% 完成率 (n=1,482)
  变体组 B: 12.8% 完成率 (n=1,476)
  p = 0.0003 ✅ 显著
  效果: +52% 相对提升
  护栏: 垃圾注册率 +5%（不显著），客服工单 -2%

决策: ✅ 全量发布
```

### 示例 2：定价页布局实验

```yaml
实验名称: 定价页从 3 列改为高亮中间推荐方案
实验 ID: EXP-20260901-002
实验类型: A/B Test

问题定义:
  网站定价页目前是 3 列等宽布局，所有方案视觉权重相同。
  大部分用户选择了最便宜的 Basic 方案（62%），
  Pro 方案仅占 23%，Enterprise 占 15%。
  Pro 方案是 ARPU 最高且留存最佳的方案（ARPU $49 vs Basic $19），
  需要引导更多用户选择 Pro。

假设:
  如果我们将定价页改为「高亮推荐方案」布局（Pro 方案放大 + 
  "Most Popular"标签 + 颜色区分）
  那么 Pro 方案的选择比例会从 23% 提升到 35% 以上
  因为锚定效应和视觉引导会使用户更倾向"推荐"选项
  用 subscription_started 事件的 plan_tier 属性分布来衡量

实验设计:
  控制组 A (50%): 现有 3 列等宽布局
  变体组 B (50%): 高亮推荐布局
    - Pro 方案居中，放大 120%
    - 添加 "🔥 Most Popular" 徽章
    - 核心功能描述加粗
    - Basic 和 Enterprise 保持原样但略微变灰

目标指标:
  主要指标: Pro 方案选择占比
  次要指标:
    - 整体购买转化率（不能因为引导而降低总量）
    - 平均订单价值 (AOV)
    - 30 天降级率（买了 Pro 是否会降级）
  护栏指标:
    - 整体购买转化率（不能降低超过 5%）
    - 页面跳出率（不能上升超过 10%）

样本量与时长:
  基线: Pro 占比 23%
  期望最小提升: +52% (23% → 35%)
  所需样本量: ~800 购买用户
  日购买量: ~40
  预计时长: 21-28 天

决策: 待实验完成后判断
```
